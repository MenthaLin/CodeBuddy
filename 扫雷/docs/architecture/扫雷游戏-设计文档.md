### 1.1 架构说明

- **单页应用 (SPA)**：前端使用原生 JavaScript ES Module，无需构建工具
- **BFF 模式**：Express 服务器作为 BFF (Backend For Frontend)，封装 Supabase 操作
- **数据库**：Supabase PostgreSQL，通过 `@supabase/supabase-js` 访问
- **通信**：前端通过 `fetch` 调用 Express API，Express 通过 Supabase SDK 访问数据库

### 2.3 游戏状态机

```
                    ┌─────────┐
                    │  idle   │ ← 初始状态 / 新游戏
                    └────┬────┘
                         │ 首次点击 (触发地雷布局)
                         ▼
                    ┌─────────┐
          ┌────────│ playing │────────┐
          │        └────┬────┘        │
          │ 暂停         │             │
          ▼             │             ▼
    ┌─────────┐         │       ┌─────────┐
    │ paused  │─────────┘       │  lost   │ ← 点到地雷
    └─────────┘ 恢复            └─────────┘
                                     │
                                     │ 新游戏
                                     ▼
                                ┌─────────┐
                                │  won    │ ← 所有安全格打开
                                └─────────┘
```

状态转换规则：
- `idle → playing`：首次左键点击（触发 `firstClickDone`，布设地雷确保安全首点）
- `playing → paused`：点击暂停按钮
- `paused → playing`：点击恢复按钮
- `playing → lost`：点到地雷
- `playing → won`：所有非地雷格均已打开
- `won/lost → idle`：点击新游戏

## 3. 数据库设计

### 3.1 表结构

```sql
-- 用户表
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  nickname VARCHAR(50) NOT NULL UNIQUE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 游戏记录表
CREATE TABLE game_records (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  difficulty VARCHAR(10) NOT NULL CHECK (difficulty IN ('easy', 'medium', 'hard')),
  result VARCHAR(10) NOT NULL CHECK (result IN ('win', 'loss')),
  time_seconds INTEGER NOT NULL CHECK (time_seconds >= 0),
  completed_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 3.2 索引

```sql
-- 加速用户游戏记录查询
CREATE INDEX idx_game_records_user_id ON game_records(user_id);

-- 加速排行榜查询（按难度、结果、时间排序）
CREATE INDEX idx_game_records_leaderboard ON game_records(difficulty, result, time_seconds)
  WHERE result = 'win';

-- 加速用户统计
CREATE INDEX idx_game_records_user_stats ON game_records(user_id, difficulty, result);
```

### 3.3 RLS 策略

```sql
-- 启用 RLS
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE game_records ENABLE ROW LEVEL SECURITY;

-- users 表策略
-- 任何人可以读取所有用户（排行榜需要）
CREATE POLICY "users_read_all" ON users FOR SELECT USING (true);
-- 用户可以更新自己的昵称
CREATE POLICY "users_update_own" ON users FOR UPDATE USING (auth.uid() = id);
-- 任何人都可以创建用户（注册）
CREATE POLICY "users_insert_all" ON users FOR INSERT WITH CHECK (true);

-- game_records 表策略
-- 用户只能读取自己的记录
CREATE POLICY "records_read_own" ON game_records FOR SELECT
  USING (auth.uid() = user_id);
-- 排行榜公开读取（仅胜利记录）
CREATE POLICY "records_read_public_wins" ON game_records FOR SELECT
  USING (result = 'win');
-- 用户只能插入自己的记录
CREATE POLICY "records_insert_own" ON game_records FOR INSERT
  WITH CHECK (auth.uid() = user_id);
-- 禁止修改和删除记录
```

### 3.4 用户认证方案

使用 Supabase Auth 的匿名登录 + 昵称：
1. 前端调用 `supabase.auth.signInAnonymously()` 获取匿名用户
2. 用户输入昵称后，后端创建/查找 `users` 表记录
3. 匿名用户的 `uid` 作为 `users.id`，确保数据隔离
4. 昵称唯一性在后端校验

## 5. 前端数据流设计

### 5.1 事件流

```
用户点击格子
  → Board.handleClick(row, col)
    → Game.reveal(row, col)
      → 检查是否首次点击 → 布设地雷
      → 检查是否是地雷 → Game.gameOver('lost')
      → Flood Fill 展开
      → 检查胜利条件 → Game.gameOver('won')
    → Board.updateCell(row, col, cellState)
    → Timer.start() / Timer.stop()
    → 游戏结束时 → api.submitRecord()
```

### 5.2 状态管理

采用单一状态对象，集中在 `Game` 类中：

```javascript
class Game {
  constructor(difficulty) {
    this.state = 'idle';       // idle | playing | paused | won | lost
    this.rows = 0;
    this.cols = 0;
    this.mineCount = 0;
    this.flagsPlaced = 0;
    this.board = [];           // Cell[][]
    this.startTime = null;
    this.elapsedTime = 0;
    this.firstClickDone = false;
  }
}
```

### 5.3 用户状态流

```
页面加载
  → storage.getUser() 检查 localStorage 是否有缓存用户
  → 有缓存 → 恢复用户状态
  → 无缓存 → 显示登录弹窗
  → 用户输入昵称 → api.login(nickname)
  → 登录成功 → storage.setUser(user) → 更新UI
```

## 6. 核心算法设计

### 6.1 地雷布局（安全首点）

```
function placeMines(board, mineCount, safeRow, safeCol):
  1. 收集所有可用位置（排除 (safeRow, safeCol) 及其周围8格）
  2. Fisher-Yates 洗牌可用位置列表
  3. 取前 mineCount 个位置放置地雷
  4. 计算每个格子的 adjacentMines 值
```

### 6.2 Flood Fill 展开

```
function floodFill(board, row, col):
  1. 如果越界或已揭晓或已标记 → 返回
  2. 揭晓当前格子
  3. 如果 adjacentMines === 0:
     对周围8格递归调用 floodFill
```

### 6.3 胜利检测

```
function checkWin(board):
  遍历所有格子:
    如果 !cell.isMine && !cell.isRevealed → 返回 false
  返回 true
```

### 6.4 双击快速展开 (P1)

```
function chordReveal(board, row, col):
  前提: 格子已揭晓 && adjacentMines > 0
  统计周围8格中已标记旗帜数
  如果 旗帜数 === adjacentMines:
    对周围8格中未标记且未揭晓的格子执行 reveal()
```

## 7. 安全设计

### 7.1 输入校验

- **昵称**：长度 1-50 字符，仅允许中文、英文、数字、下划线
- **难度参数**：仅允许 `easy`、`medium`、`hard`
- **游戏时间**：0-99999 秒
- **分页参数**：limit 最大 100

### 7.2 Supabase RLS

- 用户只能读取自己的游戏记录
- 排行榜仅公开胜利记录
- 禁止客户端直接操作数据库（通过 Express BFF）

### 7.3 防作弊思路

- 游戏时间由前端计时，后端校验合理性（如 easy 模式 < 1 秒视为异常）
- 后端记录请求来源 IP，防止批量刷榜
- 排行榜缓存（Redis 或内存缓存），减少数据库压力

### 7.4 Express 安全

- 使用 `helmet` 中间件设置安全 HTTP 头
- 使用 `cors` 限制允许的来源
- 使用 `express-rate-limit` 限制 API 请求频率
- 请求体大小限制 `express.json({ limit: '1mb' })`

## 11. 关键技术决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 前端框架 | 原生 JS | 轻量，无构建依赖，符合需求 |

| 数据库 | Supabase | 内置 Auth + RLS，PostgreSQL 兼容 |
| 模块化 | ES Module | 现代浏览器原生支持 |
| 状态管理 | 类实例 | 简单直接，适合单一游戏状态 |
| 音效 | Web Audio API | 无需外部音频文件，程序化生成 |
